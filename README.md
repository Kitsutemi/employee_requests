# Учёт заявок сотрудников

Веб-приложение на Python (FastAPI + SQLAlchemy + SQLite) для учёта заявок сотрудников с контролем бизнес-правил перехода статусов, фильтрацией, отчётностью и тестированием производительности на большом объёме данных.

## Быстрый старт

### 1. Установка зависимостей

```bash
cd "m:\Projects\employee requests"
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Генерация тестовых данных (опционально)

> База `app.db` уже может содержать 1 000 000 заявок и 1 000 сотрудников.  
> Для пересоздания данных (займёт несколько минут):

```bash
python generate_data.py
```

### 3. Запуск приложения

```bash
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
```

Откройте в браузере: http://127.0.0.1:8000

### 4. Замер производительности запроса

```bash
python measure_query.py
```

## Бизнес-процесс

Заявка проходит три этапа:

```
Новая → В работе → Выполнена
```

| Статус (код)   | Отображение |
|----------------|-------------|
| `new`          | Новая       |
| `in_progress`  | В работе    |
| `done`         | Выполнена   |

---

## Бизнес-правила

1. **Переходы статусов** — только последовательные:
   - `new` → `in_progress`
   - `in_progress` → `done`
   - Прямой переход `new` → `done` **запрещён**
   - Из `done` переходы невозможны

2. **Смена исполнителя** — разрешена для заявок в статусах «Новая» и «В работе»; для «Выполнена» — запрещена.

3. **Просроченная заявка** — срок выполнения (`deadline`) меньше текущей даты и статус не «Выполнена».

Правила реализованы в доменном слое: `app/domain/rules.py`, `app/domain/entities.py`.

## Структура базы данных

### Таблица `employees`

| Поле         | Тип          | Описание        |
|--------------|--------------|-----------------|
| id           | INTEGER PK   | Идентификатор   |
| full_name    | VARCHAR(200) | ФИО             |
| department   | VARCHAR(100) | Подразделение   |
| position     | VARCHAR(100) | Должность       |

### Таблица `requests`

| Поле         | Тип          | Описание                          |
|--------------|--------------|-----------------------------------|
| id           | INTEGER PK   | Идентификатор                     |
| number       | VARCHAR(20)  | Номер заявки (уникальный)         |
| created_at   | DATETIME     | Дата создания                     |
| author_id    | FK → employees | Автор                           |
| executor_id  | FK → employees | Исполнитель                     |
| description  | TEXT         | Описание                          |
| deadline     | DATETIME     | Срок выполнения                   |
| status       | VARCHAR(20)  | Статус (CHECK-ограничение)        |

### Принятые решения по проектированию

1. **Нормализация (3НФ)** — ФИО, подразделение и должность хранятся только в `employees`; в заявках — внешние ключи `author_id` и `executor_id`. Это исключает дублирование данных сотрудника.

2. **Справочник статусов** — три значения зафиксированы CHECK-ограничением на уровне БД; бизнес-логика переходов — в коде (`StatusTransitionRules`).

3. **Связи** — две FK на одну таблицу (`author`, `executor`) через SQLAlchemy `relationship` с явным указанием `foreign_keys`.

4. **Индекс для производительности** — составной индекс `(executor_id, status, deadline)` покрывает фильтрацию и сортировку целевого запроса.

## Функциональность

### Сотрудники (`/employees`)
- Просмотр справочника
- Добавление и удаление

### Заявки (`/`)
- Создание заявки (`/requests/new`)
- Фильтрация: статус, исполнитель, подразделение, просроченные
- Пагинация (50 записей на страницу)
- Смена статуса и исполнителя с проверкой правил

### Отчёты (`/reports`)
- Количество заявок по каждому статусу
- Количество просроченных заявок
- Количество выполненных заявок по каждому исполнителю

### REST API
- `POST /api/requests/` — создание
- `PUT /api/requests/{id}/status` — смена статуса
- `PUT /api/requests/{id}/executor` — смена исполнителя
- `GET /api/requests/` — список с фильтрами

## Оптимизация производительности

### Целевой запрос

```sql
SELECT * FROM requests
WHERE executor_id = ?
  AND status = 'in_progress'
  AND deadline < datetime('now')
ORDER BY deadline;
```

### Метод оптимизации

Создан **составной индекс**:

```sql
CREATE INDEX idx_requests_executor_status_deadline
ON requests (executor_id, status, deadline);
```

### Почему это ускоряет выполнение

1. **Фильтрация** — SQLite использует индекс для быстрого поиска записей по `executor_id` и `status` вместо полного сканирования 1M строк.
2. **Сортировка** — поле `deadline` включено в индекс; результаты уже упорядочены, отдельная операция SORT не требуется.
3. **ANALYZE** — после создания индекса обновляется статистика для оптимизатора запросов.

### Результаты замеров

Запустите `python measure_query.py` для актуальных цифр на вашей машине.

Пример результата на базе **1 000 000 заявок** (исполнитель ID=1, 295 записей):

| Этап              | План выполнения (EXPLAIN)                                      | Среднее время |
|-------------------|----------------------------------------------------------------|---------------|
| Без индекса       | `SCAN requests` + `USE TEMP B-TREE FOR ORDER BY`               | **0.2120 с**  |
| С индексом        | `SEARCH requests USING INDEX idx_requests_executor_status_deadline` | **0.0009 с**  |
| **Ускорение**     |                                                                | **~234x**     |

## Дополнительное задание: изменения при расширении системы

### 1. Согласование руководителем

- Добавить статус `pending_approval` («На согласовании») между «Новая» и «В работе»
- Таблица `approvals` (request_id, approver_id, decision, comment, decided_at)
- Расширить `StatusTransitionRules`: `new → pending_approval → in_progress`
- Уведомления руководителю (email/внутренние)
- UI: очередь заявок на согласование для руководителя подразделения

### 2. Несколько исполнителей

- Таблица связи `request_executors` (request_id, employee_id, role, assigned_at)
- Убрать `executor_id` из `requests` или оставить как «ответственный»
- Бизнес-правила: завершение заявки только когда все исполнители отметили выполнение, либо по правилу «любой/все»

### 3. История изменения статусов

- Таблица `request_status_history` (id, request_id, old_status, new_status, changed_by, changed_at, comment)
- Триггер или сервисный слой записывает каждый переход
- UI: вкладка «История» на карточке заявки

### 4. Сроки зависят от типа заявки

- Справочник `request_types` (id, name, default_deadline_days, priority)
- Поле `type_id` в `requests`
- При создании: `deadline = created_at + type.default_deadline_days`
- Возможность переопределения срока с обоснованием

### 5. Разграничение прав доступа

| Роль           | Права                                              |
|----------------|----------------------------------------------------|
| Сотрудник      | Создание заявок, просмотр своих                    |
| Руководитель   | Согласование, просмотр заявок подразделения        |
| Администратор  | Полный доступ, управление справочниками            |

- Таблицы `users`, `roles`, `user_roles`
- Связь `users.employee_id → employees.id`
- Middleware авторизации (JWT / session)
- Фильтрация запросов по роли на уровне сервисов
